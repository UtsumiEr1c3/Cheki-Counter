## Context

当前活动模型由 `events(name, venue, date, created_at)` 表示,切奇花费来自 `records.subtotal` 按 `event_id` 聚合。偶活总览已经有按年份筛选、电切活动过滤、活动卡片和底部汇总,但金额只覆盖切奇,无法表达每场的门票成本。

门票价是活动级事实:同一场活动可以有多条切奇记录或没有切奇记录,但门票只应计一次。用户也可能通过三条路径创建活动:新建活动、添加切奇时输入活动名、新建偶像首条切奇时输入活动名。

## Goals / Non-Goals

**Goals:**

- 为每个活动持久化一个非负整数门票价。
- 在所有会创建或选择活动的手动入口中支持填写、带出和保存门票价。
- 在偶活总览和活动详情中同时展示门票价、切奇价和合计价。
- 扩展 CSV 导入导出,让门票价可备份、迁移,并兼容旧 CSV。

**Non-Goals:**

- 不支持一场活动多人票、分摊票价、折扣明细或交通餐饮等其它费用。
- 不新增活动编辑/删除 UI。
- 不改变现有电切活动整场隐藏规则。
- 不改变偶像总金额统计;偶像金额仍只来自切奇 `records.subtotal`。

## Decisions

### D1. 门票价存放在 `events.ticket_price`

为什么:门票属于活动场次,不是某个切奇记录。放在 `events` 上能让纯打卡活动也有门票价,并避免同场多条 records 时重复统计。

候选对比:

- `events.ticket_price`: 聚合简单,和总览页面的数据边界一致。
- `records.ticket_price`: 会在同场多偶像时重复计入,纯打卡活动也无法记录门票。
- 独立 `event_costs` 表:未来扩展性更强,但当前只有一个门票价字段,复杂度过高。

### D2. 门票价为非负整数,默认 `0`

为什么:现有切奇单价、小计都以整数日元/人民币金额保存,门票价沿用同一模型。`0` 表示未记录或免费票,能无痛迁移老 DB 与旧 CSV。

候选对比:

- `INTEGER NOT NULL DEFAULT 0`: 查询和 UI 简单,无 null 分支。
- `INTEGER NULL`: 可区分未知与免费,但总览汇总和 CSV 兼容需要更多状态表达。
- `REAL`: 当前金额体系没有小数输入需求,引入浮点会带来格式和精度负担。

### D3. upsert 活动时允许补写门票价

为什么:用户可能先创建活动时没填门票,之后添加切奇时才补上;也可能选择已有活动时自动带出票价。规则采用保守更新:三元组已存在且旧 `ticket_price = 0`、新输入 `> 0` 时更新为新值;旧值已大于 0 时保持旧值,避免意外覆盖。

候选对比:

- 仅首次创建写入:实现最简单,但漏填后无编辑 UI,用户无法补救。
- 每次 upsert 都覆盖:方便修正,但添加新记录时容易误改历史活动门票。
- 仅从 0 补写为正数:符合当前无编辑 UI 的约束,也降低误覆盖风险。

### D4. 汇总对象增加三种金额

为什么:总览页面需要同时显示门票、切奇、合计。`EventWithSummary` 应携带 `ticketPrice`,保留现有 `totalAmount` 作为切奇总价,并提供 `grandAmount = ticketPrice + totalAmount` 这样的派生语义。

候选对比:

- Repository 返回原子金额,UI 求和:清晰,避免 SQL 里混合 UI 表达。
- SQL 直接返回 grand_total:可少写一行 Dart,但让字段含义更隐蔽。

### D5. CSV 新增第 14 列 `门票价格`

为什么:现有 CSV 已有活动三列和电切列,门票价格和活动强绑定,追加在末尾能保持旧列索引最大程度稳定。旧 9/11/12/13 列继续按原规则导入,缺门票列默认为 `0`。

候选对比:

- 追加第 14 列:兼容性最好。
- 插入到活动三列旁边:可读性略强,但会移动 `电切` 列,破坏当前 13 列导入假设。

## Risks / Trade-offs

- [Risk] 没有活动编辑 UI,用户填错非零门票后无法直接修改 → Mitigation: 本 change 不扩大编辑范围;可在后续单独提活动编辑 change。当前只允许 0 补写为正数,避免误覆盖。
- [Risk] 总览底部金额文案变长,小屏可能拥挤 → Mitigation: 使用短标签 `门票 / 切 / 合计`,必要时允许 Wrap 或多行居中展示。
- [Risk] CSV 同一活动多行携带不同门票价 → Mitigation: 导入复用 D3 规则,已存在非零门票不被后续行覆盖;0 可被正数补写。
- [Risk] 电切活动被总览隐藏后其门票价是否计入总览可能有歧义 → Mitigation: 沿用现有规则,被隐藏的 event 不计入总览活动数、门票总价、切奇总价或合计。

## Migration Plan

1. 数据库版本从 3 升到 4。
2. `_onCreate` 创建 `events.ticket_price INTEGER NOT NULL DEFAULT 0`。
3. `_onUpgrade` 在 `oldVersion < 4` 时执行 `ALTER TABLE events ADD COLUMN ticket_price INTEGER NOT NULL DEFAULT 0`。
4. 应用层所有读取 `events` 的 SQL 显式包含 `ticket_price`。
5. 回滚到旧版本 App 时旧代码不会读取该列,但若旧版本重新创建 DB 则不保留门票数据;不提供自动降级迁移。
