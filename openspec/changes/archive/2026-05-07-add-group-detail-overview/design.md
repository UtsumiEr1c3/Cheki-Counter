## Context

当前 `GroupOverviewPage` 只展示团体级全量聚合数据,数据来自 `IdolRepository.getGroupAggregates()`。用户已经可以从首页偶像卡片进入 `IdolDetailPage`,也可以在统计页用年份筛选和"按切数 / 按金额"切换查看全局排行。

这次变更补齐团体统计的年份视角和下钻路径:用户先在团体总览按年份查看团体排行,再点击某个团体进入该团体的小偶像聚合列表,并能继续用年份和排序模式缩小视角。

## Goals / Non-Goals

**Goals:**

- 团体总览项可点击,进入对应团体详情页。
- 团体总览页支持"全部 / 年份"筛选,年份来自现有记录日期。
- 团体详情页默认继承团体总览页当前选择的年份。
- 团体详情页展示该团体汇总和小偶像列表。
- 团体详情页支持"全部 / 年份"筛选,年份来自现有记录日期。
- 团体详情页支持"按切数 / 按金额"排序切换。
- 小偶像行复用现有小偶像详情导航。

**Non-Goals:**

- 不修改数据库 schema。
- 不新增 CSV 列或改变导入导出格式。
- 不支持编辑团体名、小偶像名或应援色。
- 不在本次变更中新增团体图表。

## Decisions

### D1. 新增团体详情页,不在总览页内展开

**决定**:点击团体项后 push 一个独立的 `GroupDetailPage`。

**为什么**:团体详情需要汇总栏、年份筛选、排序切换和成员列表,信息密度已经超过一张 `ListTile` 原地展开的舒适范围。独立页面也能保留 AppBar 标题和返回行为,与现有 `EventsOverviewPage -> EventDetailPage` 的模式一致。

**候选对比**:

- 原地展开:少一个页面,但滚动位置、筛选控件和长列表会让总览页变复杂。
- 独立页面:多一个文件,但导航语义清晰,后续也方便扩展图表或分组统计。

### D2. 数据查询放在 `IdolRepository`

**决定**:扩展 `IdolRepository.getGroupAggregates()` 支持可选 `year`,并增加按 `group_name`、可选 `year`、`sortBy` 查询偶像聚合的方法。

**为什么**:`IdolRepository` 已经负责偶像与团体聚合,这两个查询仍然围绕 `idols + records` 的聚合结果,放在同一仓库最贴合当前分层。总览和详情使用同一个年份口径,能避免"总览 2026,详情全部"的错位感。

**候选对比**:

- 新增 `StatisticsRepository`:概念更细,但当前项目尚无这个仓库,为一个查询引入新抽象偏重。
- 放在页面里 raw SQL:实现短,但破坏 data/features 分层。

### D3. 年份列表复用 `RecordRepository.getDistinctYears()`

**决定**:团体总览页和团体详情页的年份下拉都使用全局实际记录年份,额外包含"全部"。团体详情页接收从总览页传入的初始年份,用户仍可在详情页改选。

**为什么**:统计页已有同样的年份筛选模型,复用能保持 UI 和数据语义一致。选择某年后,团体总览只统计 `records.date` 属于该自然年的记录;没有该年份记录的团体不出现在列表中。进入详情后,成员列表沿用同一个年份筛选;没有该团体记录的小偶像不出现在该年列表中。

**候选对比**:

- 只显示该团体实际出现过的年份:下拉更短,但需要新增团体专属年份查询。
- 复用全局年份:实现简单,与统计页一致;若某年该团体无数据,页面展示空态。

### D4. 排序模式沿用统计页语义

**决定**:`sortBy = count` 时按 `SUM(records.count)` 降序;`sortBy = amount` 时按 `SUM(records.subtotal)` 降序。

**为什么**:用户明确希望也提供切换功能,而统计页已有这两个指标的心智模型。团体详情页不引入第三种排序,减少操作复杂度。

## Risks / Trade-offs

- [Risk] 年份下拉复用全局年份时,用户可能在详情页选择到该团体没有记录的年份 -> Mitigation:详情页显示清楚的空态,保留下拉和切换控件便于切回。
- [Risk] `group_name` 是用户输入文本,SQL 拼接会有注入和转义风险 -> Mitigation:新增查询 MUST 使用参数化 `whereArgs` 或 rawQuery 参数。
- [Risk] 团体详情页与统计页有相似的筛选/排序控件,后续可能重复 -> Mitigation:本次保持局部实现,等第三处复用出现再抽组件。
- [Risk] 总览页年份和详情页年份口径不一致会造成误读 -> Mitigation:详情页默认接收总览页当前年份,并且汇总栏直接从当前成员列表聚合。
